What BugOff does

Four questions, answered instantly.

⚡

Which bugs matter most?

Weighted scoring across severity, bounty value, reproductions, and age — normalized to a 0-100 scale.

🏆

What should I fix first?

Top-K retrieval from a max-heap in O(log n). No more scrolling through noisy issue lists.

🔄

What changes after a fix?

Fix simulation removes bugs from the heap and rebalances — see how the priority landscape shifts live.

📅

Best use of my time?

Greedy scheduler maximizes total priority within your developer-hour budget. Like a knapsack solver for bugs.

🧠

AI explanations

Optional AI layer explains why a bug ranks highly and narrates fix plans. Falls back to templates if unavailable.

📊

Greedy vs optimal

Brute-force comparison validates the greedy approach on small datasets — see exactly how close it gets.

How it works

1

Fetch real bugs

Pulls live issues from OWASP BLT via API. Falls back to GitHub issues if BLT is unreachable. Real data, not mocks.

2

Score each bug

Each bug is scored with a weighted formula: Severity×0.4 + Bounty×0.3 + Repro×0.2 + Age×0.1. All inputs normalized to 0-100.

3

Build a max-heap

Scored bugs are inserted into a max-heap using Go's container/heap. Top-priority bug is always O(1) to access.

4

Schedule fixes

Given a time budget (e.g. 8 hours), the greedy scheduler picks bugs by priority / fix_hours — maximizing impact per hour.

5

Deploy and iterate

Fix a bug → heap rebalances → new priorities emerge. Keep deploying until your budget runs out or the city is safe.

Scoring formula

Priority =
  (Severity × 0.4) +
  (BountyValue × 0.3) +
  (Reproductions × 0.2) +
  (Age × 0.1)
